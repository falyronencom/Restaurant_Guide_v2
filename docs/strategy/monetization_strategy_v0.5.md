## 1. Философия платформы

### 1.1 Основной принцип

Честная выдача + монетизация через инструменты, а не через искажение ранжирования. Nirivio не продаёт позиции в выдаче. Все сортировки (расстояние, рейтинг, цена) отражают реальные данные без платных бустов. Монетизация строится на инструментах привлечения и конверсии, которые создают ценность и для партнёра, и для пользователя.

### 1.2 Структурное преимущество

Nirivio создано силами распределённого интеллекта (координатор + AI-агенты) без большого штата сотрудников. Для сравнения: ARTOX (Relax.by) содержит 250+ человек. Операционные расходы Nirivio на порядки ниже, что позволяет предлагать щедрый бесплатный уровень без искусственных ограничений, устанавливать доступные цены на платные инструменты и экспериментировать быстро без бюрократических цепочек. Порог рентабельности радикально ниже: где конкуренту нужны сотни платящих партнёров, Nirivio достаточно десятков.

### 1.3 Позиционирование vs конкуренты

Relax.by строит монетизацию на искусственном ограничении бесплатного уровня: заведение теряется на дальних страницах поиска, на его карточке рекламируются конкуренты (блоки «Похожие места»), нет бронирования, ограничены фотографии, нет аналитики. По сути бесплатное размещение — это «визитка-пустышка». Платный тариф снимает эти искусственные ограничения. Модель работает, но создаёт негатив: Relax.by сначала создаёт проблему (невидимость), а потом продаёт решение (видимость).

Nirivio предлагает обратный подход. Бесплатный уровень — полноценное присутствие без ограничений. На карточке заведения никогда не рекламируются конкуренты. Ранжирование честное и прозрачное. Платный уровень — не "разблокировка" урезанных функций, а дополнительные инструменты привлечения клиентов (акции, бронирование). Позиционирование для партнёра: «Мы просим подписку для поддержания и развития проекта — чем известнее платформа, тем больше клиентов видят вашу карточку.»

## 2. Ранжирование и сортировка

### 2.1 Основная философия: шаговая доступность

Ключевое отличие Nirivio — фокус на шаговой доступности. Пользователь открывает приложение, чтобы найти место рядом. При сортировке по расстоянию порядок строго математический: заведение в 100м всегда выше заведения в 900м. Никакие факторы не нарушают этот порядок. Аналогично для рейтинга (реальные отзывы с учётом достоверности) и цены (реальный средний чек).

**Дефолтная сортировка зависит от контекста:**

| Контекст | Дефолтная сортировка | Обоснование |
|----------|---------------------|-------------|
| Геолокация включена | **По расстоянию** | Философия шаговой доступности: пользователь открыл приложение → видит ближайшее |
| Геолокация выключена | **По рейтингу** (weighted) | Без координат "что рядом" невозможно → browsing каталога города, рейтинг полезнее |

Пользователь с геолокацией видит ближайшее по умолчанию. Хочет лучшее в городе? Переключает на "По рейтингу" — сознательное действие, один тап в фильтре. Два режима отвечают на разные вопросы: "Что рядом?" (расстояние) vs "Что лучшее?" (рейтинг) — и не должны смешиваться в основном критерии.

### 2.2 Текущая реализация (по данным Discovery Report)

Поиск реализован в `searchService.js` с функцией `buildOrderByClause()` (строка 122). Расстояние рассчитывается через PostGIS `ST_Distance` непосредственно в SQL-запросе, что обеспечивает математическую точность.

Текущие режимы сортировки и тайбрейкеры:

| Сортировка | Основной критерий | Тайбрейкер 1 | Тайбрейкер 2 |
|---|---|---|---|
| По рейтингу (default) | average_rating DESC | review_count DESC | name ASC |
| По расстоянию | distance_km ASC | average_rating DESC | name ASC |
| По цене (возр.) | price_range ASC (через CASE $→1, $$→2, $$$→3, $$$$→4) | average_rating DESC | name ASC |
| По цене (убыв.) | price_range DESC | average_rating DESC | name ASC |

Финальный тайбрейкер `name ASC` обеспечивает детерминированный порядок (стабильная выдача без "скачков"), но не несёт пользовательской ценности.

### 2.3 Целевые изменения сортировки и тайбрейкеров

**С геолокацией (координаты доступны):**

| Сортировка | Основной критерий | Тайбрейкер 1 | Тайбрейкер 2 | Тайбрейкер 3 |
|---|---|---|---|---|
| По расстоянию (дефолт) | distance_km ASC | base_score DESC (completeness) | weighted_rating DESC | name ASC |
| По рейтингу | weighted_rating DESC | review_count DESC | distance_km ASC | name ASC |
| По цене | price_range через CASE | distance_km ASC | weighted_rating DESC | name ASC |

**Без геолокации (§2.8):**

| Сортировка | Основной критерий | Тайбрейкер 1 | Тайбрейкер 2 | Тайбрейкер 3 |
|---|---|---|---|---|
| По рейтингу (дефолт) | weighted_rating DESC | review_count DESC | base_score DESC | name ASC |
| По цене | price_range через CASE | weighted_rating DESC | review_count DESC | name ASC |
| По расстоянию | *Недоступна* — нет координат, нечего считать | — | — | — |

`weighted_rating` — байесовский средний (§2.5), заменяет `average_rating + twoTierRating` во всех контекстах.

Логика: при прочих равных, пользователю полезнее увидеть ближайшее (сортировка по рейтингу/цене → расстояние как тайбрейкер) или более информативное (сортировка по расстоянию → completeness score как тайбрейкер).

### 2.4 Исправление: NULL price_range в сортировке по цене

**Выявлено при Code-верификации.** Текущий код использует `ELSE 0` в CASE-выражении для price_range. При `price_asc` заведения без цены получают значение 0 и оказываются **перед** самыми дешёвыми (`$` = 1). Это противоречит логике: пользователь ищет дешёвое, а видит сначала карточки без цены.

**Решение:** `ELSE 5` вместо `ELSE 0`. Заведения без price_range уходят в конец при обоих направлениях сортировки (после `$$$$` = 4). Это согласуется с completeness score: неполная карточка проигрывает, а не выигрывает.

**Для Implementer:** изменить `ELSE 0` → `ELSE 5` в `buildOrderByClause()`, применить ко всем трём query paths.

### 2.5 Баланс рейтинга и достоверности: байесовский средний

**Проблема.** Заведение с 1 отзывом на 5.0 формально имеет рейтинг выше, чем заведение с 4.8 и 200 отзывами. Первоначальное решение — двухэшелонная сортировка (порог 3 отзыва) — создало обратный перекос: 3.0★×4 отзыва показывалось выше 5.0★×2 отзыва. Жёсткая стена между эшелонами игнорировала разницу в 2 звезды.

**Решение: байесовский средний (weighted rating).**

```
weighted_rating = (v × R + m × C) / (v + m)
```

Где:
- `v` = количество отзывов заведения (`review_count`)
- `R` = средний рейтинг заведения (`average_rating`)
- `m` = **5** — порог доверия (сколько "виртуальных" отзывов со средним рейтингом подмешивается)
- `C` = **3.5** — консервативная презумпция ("пока не доказано обратное, считаем чуть ниже среднего")

**Интуиция:** до набора `m` отзывов рейтинг "притягивается" к среднему. Чем больше реальных отзывов — тем ближе к фактическому рейтингу. Формула плавная, без жёстких стен и порогов.

**Проверка на реальных кейсах (m=5, C=3.5):**

| Заведение | Рейтинг | Отзывы | Weighted | Позиция |
|-----------|---------|--------|----------|---------|
| Проверенное отличное | 4.8★ | 200 | **4.77** | 1 — заслуженно первое |
| Хорошее популярное | 4.5★ | 50 | **4.41** | 2 |
| Новое отличное (3 отзыва) | 5.0★ | 3 | **4.06** | 3 — высокое, но не обгоняет проверенных |
| Новое отличное (2 отзыва) | 5.0★ | 2 | **3.93** | 4 |
| Один отзыв 5.0 | 5.0★ | 1 | **3.75** | 5 — не первое место |
| Плохое с отзывами | 3.0★ | 4 | **3.28** | 6 — ниже 5.0×2, что справедливо |
| Нет отзывов | 0★ | 0 | **3.50** | — (презумпция среднего) |

**Обе исходные проблемы решены:**
- 5.0★×1 (3.75) НЕ обгоняет 4.8★×200 (4.77) — защита от "новичка" ✓
- 5.0★×2 (3.93) ВЫШЕ 3.0★×4 (3.28) — нет абсурда двухэшелонной стены ✓

**SQL-реализация — одна строка:**

```sql
ORDER BY
  (e.review_count * e.average_rating + 5 * 3.5) / (e.review_count + 5) DESC,
  e.review_count DESC,
  distance_km ASC,  -- или base_score DESC без геолокации
  e.name ASC
```

**Для Implementer:**

```javascript
const BAYESIAN_PRIOR_COUNT = 5;     // m — порог доверия
const BAYESIAN_PRIOR_RATING = 3.5;  // C — консервативная презумпция

const weightedRating = `(e.review_count * e.average_rating + ${BAYESIAN_PRIOR_COUNT} * ${BAYESIAN_PRIOR_RATING}) / (e.review_count + ${BAYESIAN_PRIOR_COUNT})`;
```

- Заменяет `twoTierRating` + `e.average_rating DESC` во всех контекстах
- `RATING_SORT_MIN_REVIEWS` — удалить (больше не нужна)
- Применить ко всем query paths (§2.7): buildOrderByClause, searchByBounds (inline), CTE path
- CTE path: `replaceAll('e.', 'ne.')` работает — в формуле только `e.review_count` и `e.average_rating`
- Параметры m и C — конфигурируемые константы, тюнятся без миграции

**Выбор параметров:**

`m=5`: при 92 заведениях — оптимальный баланс. Заведение с 5 отзывами "на полпути" к реальному рейтингу, с 15 — на 75%. Стандарт индустрии (IMDB, Amazon, Yelp используют аналогичный подход).

`C=3.5`: не средний рейтинг по платформе (~4.2 по seed-данным), а **консервативная константа**. Причины: не меняется при изменении среднего по платформе, создаёт стимул набирать отзывы (чем больше отзывов > 3.5, тем выше weighted), защищает от завышения стартовой позиции.

**Прозрачность для пользователя:** формула работает под капотом. На карточке отображается реальный рейтинг (4.8★) + количество отзывов (см. §2.6). Пользователь сам считывает: "4.8 на 200 отзывах надёжнее, чем 5.0 на двух" — как на Booking.com.

### 2.6 Отображение review_count на карточке (паттерн Booking.com)

**Контекст:** Booking.com при сортировке по рейтингу отображает количество отзывов рядом с рейтингом. Отель с 9.8★×2847 отзывов выше отеля с 10.0★×3 отзыва — и пользователь **видит почему**. Никакого когнитивного диссонанса.

**Решение: два слоя одного решения.**

| Слой | Что делает | Где |
|------|-----------|-----|
| Backend (формула) | Определяет порядок: weighted_rating | `buildOrderByClause` — SQL |
| Frontend (UI) | Объясняет порядок пользователю | Карточка заведения — бейдж рядом с рейтингом |

Один без другого не работает: формула без отображения review_count → пользователь не понимает почему 4.8 выше 5.0. Отображение без формулы → числа видны, но порядок не соответствует логике.

**Формат отображения:**

```
★ 4.8 · 47 отзывов          — много, доверие высокое
★ 5.0 · 2 отзыва            — рейтинг выше, но отзывов мало — видно сразу
★ 4.2 · 12 отзывов          — средний диапазон
★ — · Нет отзывов            — новое заведение
```

**Для Implementer:** склонение "отзыв/отзыва/отзывов" по правилам русского языка (1 отзыв, 2-4 отзыва, 5-20 отзывов, 21 отзыв...). Стандартная функция pluralization для Flutter.

### 2.7 Архитектурное предупреждение: query paths

Discovery Report и Code-верификация выявили, что логика ранжирования должна применяться к нескольким независимым путям запросов. С учётом §2.8 (поведение без геолокации) — это четыре режима:

1. **`searchByRadius()`** — поиск с координатами пользователя (основной режим)
2. **`searchWithoutLocation()`** — поиск без координат, с фильтром по городу (fallback при отключённой геолокации)
3. **`searchByBounds()`** — поиск для карты (свой ORDER BY: `average_rating DESC, review_count DESC`)
4. **Мобильный fallback** — переключение между (1) и (2) в зависимости от наличия реальных координат

Любое изменение в сортировке или тайбрейкерах необходимо синхронно применить ко всем paths. В будущем стоит рефакторить в единый query builder, но для Horizon 2 достаточно синхронного обновления.

**Для Coordinator:** при проверке реализации тестировать сортировку в четырёх режимах: поиск с координатами, поиск без координат (город выбран), вид карты, и переключение при отключении геолокации.

### 2.8 Поведение без геолокации

**Критическая находка (Code-верификация):** текущее мобильное приложение при отключённой геолокации тихо подставляет координаты центра Минска (53.9006, 27.5590) и вызывает `searchByRadius()` как будто координаты реальные. Пользователь в Гомеле без геолокации видит расстояния от центра Минска и минские заведения в топе. Расстояния на карточках — ложные. Функция `searchWithoutLocation()` существует на бэкенде, но мобильное приложение её никогда не вызывает.

**Это баг, а не feature.** Приоритет исправления — высокий (§9, Приоритет 1).

**Контекст аудитории:** по индустриальным данным, 30-40% пользователей iOS и 15-25% Android отказывают в точной геолокации. Для приложения с фокусом на шаговую доступность — значительная аудитория.

**Целевое поведение:**

**Шаг 1 — Честность.** Не показывать фейковые расстояния. Если координаты реальные — `searchByRadius()`, расстояния на карточках. Если координат нет — `searchWithoutLocation()`, расстояния не отображаются.

**Шаг 2 — Выбор города.** При отсутствии геолокации предложить выбрать город:

```
[Геолокация отклонена]
  ↓
Экран: "Выберите город"
  → Минск | Гродно | Брест | Гомель | Витебск | Могилёв | Бобруйск
  ↓
searchWithoutLocation({ city: 'Гомель' })
  ↓
Сортировка по рейтингу — без расстояний
```

**Шаг 3 — Мягкий запрос геолокации.** После выбора города, ненавязчивый баннер один раз за сессию: "Включите геолокацию, чтобы видеть заведения в шаговой доступности" — [Включить] [Позже]. Объясняет конкретную ценность, не блокирует использование.

**Дефолтная сортировка без геолокации:** по рейтингу (Вариант A). Без координат пользователь не ищет "что рядом" — он просматривает каталог города. Рейтинг — наиболее полезный критерий для browsing. Сортировка по расстоянию недоступна (нет координат, нечего считать).

**Тайбрейкеры без геолокации:**

Где в §2.3 используется расстояние как тайбрейкер — подставляется следующий по полезности фактор:

| Сортировка | С координатами (§2.3) | Без координат |
|------------|----------------------|---------------|
| По рейтингу | TB1: review_count, TB2: расстояние, TB3: name | TB1: review_count, TB2: base_score, TB3: name |
| По цене | TB1: расстояние, TB2: rating, TB3: name | TB1: rating, TB2: review_count, TB3: name |

Сортировка по цене без геолокации полностью детерминированная: внутри каждой ценовой категории ($, $$, $$$,) заведения ранжируются по рейтингу, затем по review_count, затем по алфавиту. Никакой рандомности.

**Карта без геолокации:** `searchByBounds()` работает от viewport карты, не от координат пользователя. Карта технически функционирует. При первом открытии — центрировать на выбранном городе. Пользователь навигирует вручную.

## 3. Completeness Score

### 3.1 Назначение

Completeness score — числовой показатель полноты карточки заведения. Используется как тайбрейкер при сортировке по расстоянию (при равном расстоянии, более информативная карточка выше) и как мотиватор для партнёра (прогресс-бар в дашборде). Вес в ранжировании скромный — это вторичный фактор, а не основной.

### 3.2 Контекст: обязательные vs опциональные поля

При регистрации заведения и подаче на модерацию партнёр обязан заполнить: название, адрес и координаты, категорию и кухню, часы работы, фотографии интерьера и меню. Это философия платформы: Nirivio решает проблему информативности, поэтому базовое качество карточек обеспечивается на этапе модерации. Все active заведения уже имеют эти данные.

Completeness score считается только по реально опциональным полям:

| Поле | Приоритет для пользователя | Техническое поле в БД |
|---|---|---|
| Описание заведения | Высокий — помогает понять, что за место | `description` (TEXT, nullable) |
| Средний чек | Высокий — ключевой фильтр выбора | `price_range` (VARCHAR(4), nullable) |
| Атрибуты (Wi-Fi, парковка, и т.д.) | Высокий — практическая информация | `attributes` (JSONB, default '{}') |
| Телефон | Средний — контактная информация | `phone` (VARCHAR(20), nullable) |
| Email | Низкий — альтернативный контакт | `email` (VARCHAR(255), nullable) |
| Веб-сайт / соц-сети | Низкий — дополнительная информация | `website` (VARCHAR(255), nullable) |

### 3.3 Техническая реализация

В схеме БД существует колонка `base_score` (INTEGER, default 0), `boost_score` (INTEGER, default 0) и индекс `idx_establishments_ranking` по `(base_score + boost_score) DESC`. Ни один запрос их не использует — это "призраки схемы", заложенные на раннем этапе.

Completeness score записывается в `base_score`. Пересчитывается при каждом UPDATE на establishment. `boost_score` остаётся на нуле (философия честного ранжирования).

Принципы для Implementer: описание и price_range весят больше email и website; сумма нормализуется к фиксированному максимуму (например, 100); пустой attributes ({}) считается незаполненным; количество атрибутов внутри JSON может давать градуированный балл.

### 3.4 Для seed-карточек

Seed-карточки будут иметь естественно низкий completeness score: без описания, без price_range, с минимальными атрибутами. Claimed и обогащённые карточки поднимаются выше seed при прочих равных, стимулируя claiming.

## 4. Структура тарифов

### 4.1 Два уровня: Присутствие + Продвижение

**Бесплатный уровень — «Присутствие»**

Полноценная ценность без искусственных ограничений. Полная карточка (фото, описание, меню, часы работы). Полная аналитика (просмотры, избранное, клики). Ответы на отзывы без ограничений. Честная позиция в выдаче. Отсутствие рекламы конкурентов на карточке.

Техническое примечание: сейчас в `mediaService.js` лимиты фото привязаны к `subscription_tier` (free: 10 интерьер + 10 меню). В рамках философии "бесплатный уровень без ограничений" эти лимиты нужно пересмотреть.

**Платный уровень — «Продвижение»**

Предварительно — один тариф. **Финальное решение о количестве тарифов привязано к ценовой точке** (см. §4.4):

1. **Акции** — создание промо-предложений (скидка, спецпредложение, событие). Отдельная метрика кликов на баннер акции. Акции не влияют на позицию в основной выдаче.

2. **Бронирование столиков** — прямая конверсия из просмотра в действие. Пользователь бронирует (дата, время, гости), партнёр подтверждает/отклоняет. Входит в подписку без комиссий — контраст с Relax.by.

3. *(Будущее)* Push-уведомления пользователям из «Избранного».

### 4.2 Акции: поверхности видимости

**Согласованное решение (Code-верификация, 20 марта):**

Акции видны через две поверхности без искажения выдачи:

1. **Бейдж на карточке** в результатах поиска (список) — яркий элемент с описанием акции, позиция карточки не меняется.
2. **Бейдж на пине** на карте — тот же пин (оранжевый #F06B32), но с маленьким тегом (например, значок "%" в углу пина). Визуально привлекает, не ломает цветовую систему.

**Осознанно отложено:**

- **Фильтр "Акции" в основном поиске** — бинарный фильтр скрывает заведения без акций, что противоречит философии честной выдачи. Создаёт давление "без акций невидим" — модель, от которой Nirivio отстраивается. Может быть рассмотрен когда: аудитория достаточна (фильтр не опустошает выдачу), большинство заведений имеют возможность создавать акции, и есть данные о поведении пользователей.
- **Раздел "Акции рядом"** как отдельный экран — на раннем этапе перетянет внимание от основного поиска. Логично вписывается в будущий экран "Новости" (Horizon 3-4, когда появится контент-направление и бюджет на репортёров) как подраздел, а не самостоятельная точка навигации.

**Визуальная система (единая для списка и карты):**

| Поверхность | Без акции | С акцией |
|-------------|-----------|----------|
| Карточка в списке | Обычная | Карточка + бейдж акции |
| Пин на карте | Стандартный пин | Пин + тег "%" |

**Для будущего:** если доля пинов с бейджами превысит ~25% видимых на экране, пересмотреть визуальное выделение (сделать тоньше).

### 4.3 Три фазы внедрения

**Фаза 1 — «Всё бесплатно» (запуск → 6-12 мес.)**

Цель: наполнение базы, привлечение партнёров. Все функции бесплатны, включая акции и бронирование. Ранжирование по расстоянию с completeness score как тайбрейкером. Партнёры формируют понимание ценности платформы.

**Фаза 2 — «Один платный тариф» (6-12 мес. → 1.5-2 года)**

Цель: начало монетизации. Акции и бронирование переходят в платный тариф (решение о структуре тарифа — см. §4.4). Бесплатный уровень остаётся щедрым. Сбор данных о реальном использовании инструментов.

**Фаза 3 — «Дифференциация» (когда есть реальная конверсия)**

Цель: масштабирование. Тарифные уровни определяются данными Фазы 2. Возможные дополнения: расширенная аналитика конкурентов, push-уведомления, участие в тематических подборках, раздел акций в "Новостях".

### 4.4 Количество тарифов: зависимость от ценовой точки

**Контекст решения.** Бронирование не нужно Quick-сегменту (кофейни, пекарни, фаст-фуд) — формат "взял и ушёл". При едином тарифе Quick-сегмент платит за неиспользуемую функцию. Однако ощущение "переплаты" зависит от абсолютной цены.

Акции как самостоятельный продукт — сильный инструмент. В контексте шаговой доступности акция "кофе -40% до 11:00" работает как магнит для импульсных покупок. При наличии нескольких одноформатных заведений в одной локации акции — лучший инструмент дифференциации.

| Ценовая точка | Рекомендуемая модель | Обоснование |
|---------------|---------------------|-------------|
| Низкая (< 50 BYN/мес) | Один тариф: акции + бронирование | Сумма слишком мала для дифференциации тарифов |
| Средняя (50-100 BYN/мес) | Тестировать | A/B тест или опрос партнёров |
| Высокая (> 100 BYN/мес) | Два тарифа: "Акции" и "Акции + Бронирование" | Партнёр ожидает платить только за нужное |

**Финальное решение — в отдельной сессии по ценообразованию** с конкретными цифрами белорусского рынка.

**Архитектурный маркер (критично):** система подписок должна поддерживать добавление тарифов без миграции — тарифы хранятся как данные (таблица plans), а не как enum в коде.

### 4.5 Сетевые партнёры: подписка на несколько заведений

**Scope:** Фаза 2-3. Не требует реализации сейчас, но требует архитектурной защиты.

Два варианта покупки: подписка на одно заведение (X руб/мес) и сетевая подписка на все заведения партнёра (Y руб/мес, выгоднее при 3+ точках).

**Архитектурный маркер (критично).** Поля `subscription_tier`, `subscription_started_at`, `subscription_expires_at` находятся на уровне establishment — это правильно. При любых изменениях подписка должна оставаться на уровне establishment, а не переноситься на уровень user. Если появится `partner_subscription` — она должна агрегировать подписки карточек, а не заменять их.

## 5. Claiming: стратегия наполнения базы

### 5.1 Проблема и решение

Для разрыва замкнутого круга "мало карточек → мало пользователей → мало партнёров" используется комбинация:

**Стратегия А — Массовое присутствие.** Координатор создаёт 100-200 карточек реальных заведений Минска на основе публичных данных. Правовые аспекты использования фото и описаний — отдельная сессия.

**Стратегия Б — Таргетированный claiming.** Для 10-15 ключевых заведений: создать карточку, связаться с владельцем, предложить "забрать" готовую карточку. Pitch: «Вот ваше заведение в нашем приложении, у вас уже N просмотров — хотите управлять карточкой?»

**Стратегия приглашений: "Капельница", а не "Волна".** 5-7 приглашений в неделю (не 50 за раз). Каждый ранний партнёр — источник обратной связи. Ровная нагрузка на модерацию и поддержку.

Seed-карточки не удаляются. Они остаются с низким completeness score и проигрывают claimed карточкам при прочих равных.

### 5.2 Фазы claiming (по данным Code-верификации)

**Что есть в коде:**
- `partner_id` (UUID FK→users) — эффективно immutable, нет ни одного UPDATE в кодовой базе
- `upgradeUserToPartner()` — существует в authService.js, идемпотентна
- `partner_documents` таблица — есть (company_name, tax_id, verified, verified_by, verified_at), но нет workflow верификации
- `email_verified` / `phone_verified` — колонки в users существуют, никогда не устанавливаются
- Audit log — развёрнут, готов логировать claiming

**Что нужно добавить:** `claimed_at` (TIMESTAMP, nullable), `is_seed` (BOOLEAN, default FALSE), admin-эндпоинты для claiming.

**Фаза 1: Ручной claiming** (Horizon 2, Приоритет 1)

Масштаб: 10-15 заведений. Координатор связывается с владельцем лично.

1. Admin-эндпоинт `POST /admin/establishments/:id/claim` — меняет `partner_id`, пишет `claimed_at`, логирует в audit
2. Admin-эндпоинт `POST /admin/users/:id/upgrade-to-partner` — вызывает существующую `upgradeUserToPartner()` напрямую
3. Миграция: `claimed_at TIMESTAMP`, `is_seed BOOLEAN DEFAULT FALSE`

Workflow: Координатор общается с партнёром → партнёр регистрируется → Координатор через админку апгрейдит роль + привязывает карточку.

**Фаза 1.5: Invite-link claiming** (Horizon 2, Приоритет 2 — когда seed > 50)

Масштаб: десятки заведений. Ручной claiming становится узким местом.

1. Координатор генерирует invite-link для конкретной карточки (токен → establishment_id)
2. Отправляет ссылку владельцу (WhatsApp, email, лично)
3. Владелец переходит → регистрируется/логинится → карточка автоматически привязывается
4. Ссылка сама по себе — verification by channel (Координатор знает, кому отправил)

Таблица `establishment_invite_tokens` (не нужна полноценная `establishment_claims`).

**Claiming и верификация — независимые процессы** (см. §5.3). Invite-link не пропускает верификацию, а откладывает её.

**Фаза 3: Full self-service** (Horizon 3-4, не текущий scope)

Пользователь сам находит карточку и заявляет "это моё". Требует фактор владения (код на email домена, звонок на телефон заведения). Invite-link покрывает потребности до 500+ карточек — full self-service не блокирует рост.

### 5.3 Верификация партнёров

**Принцип: claiming и верификация — два независимых процесса.**

Claiming отвечает на вопрос "эта карточка в моём кабинете?" — invite-link + регистрация достаточно. Верификация отвечает на вопрос "ты реальный бизнес?" — УНП, документы, проверка модератором.

**Пробел с invite-link партнёрами:** партнёр по ссылке пропускает wizard (включая Legal Info). Решение — **отложенная верификация** (рекомендуемый подход):

```
Invite-link → Регистрация → Карточка сразу в кабинете
  → Баннер: "Заполните данные о компании"
  → Можно редактировать карточку без документов
  → Документы обязательны для: submit на модерацию, платные функции
```

Принцип: сначала покажи ценность (карточка в кабинете с аналитикой), потом проси данные.

**Два сценария входа партнёра:**

| Сценарий | Карточка | Документы | Когда Legal Info |
|----------|----------|-----------|-----------------|
| Самостоятельная регистрация (wizard) | Создаёт сам | Вводит на шаге 6 | Сразу |
| Invite-link claiming | Получает готовую | Баннер в кабинете | Отложенно |

**Что блокируется без верификации:**

| Действие | Без верификации | С верификацией |
|----------|----------------|----------------|
| Просмотр карточки в кабинете | Да | Да |
| Редактирование карточки | Да | Да |
| Публикация (submit → pending) | Да | Да |
| Создание акций (платный тариф) | **Нет** | Да |
| Бронирование (платный тариф) | **Нет** | Да |
| Оплата подписки | **Нет** | Да |

### 5.4 Модель верификации: совокупность данных

**УНП как единственный фактор — недостаточен.** УНП публичен (чек, стенд, egr.gov.by). Аналогично — телефон, email, юрнаименование. Набор публичных данных доказывает умение пользоваться поиском, а не владение бизнесом.

**Два уровня уверенности:**

| Уровень | Что проверяется | Когда достаточно |
|---------|----------------|-----------------|
| 1 — "Скорее всего настоящий" | Набор данных (УНП, наименование, телефон) совпадает с реестром + контакт инициирован Координатором | Фазы 1-2 (invite-link, модератор проверяет) |
| 2 — "Подтверждённый владелец" | Уровень 1 + фактор владения (код на email домена / звонок на телефон заведения) | Фаза 3 (self-service) |

**Полуавтоматическая верификация модератором** (достаточно до масштаба 1000+ карточек):

```
Партнёр вводит: company_name, tax_id (УНП), phone, email

Модератору показывается:
  ✓ УНП: 9 цифр, формат корректный
  ✓ Наименование: содержит ОПФ (ООО, ИП, ЧУП)
  ✓ Телефон: белорусский формат (+375...)
  ✓ Email: домен (корпоративный или публичный)
  ⚠ Адрес: совпадает с адресом на карточке? ДА/НЕТ
  [Ссылка: проверить в egr.gov.by]
```

Время проверки: ~30 сек/кейс. При 20 верификациях/мес — ~10 минут. Полная автоматизация (API реестра) оправдана при сотнях верификаций/мес.

## 6. Бронирование

### 6.1 Роль в продукте

Бронирование трансформирует Nirivio из каталога в инструмент действия. «Вчера через Nirivio к вам забронировали 3 столика» — самый конкретный аргумент ценности. Входит в платный тариф без комиссий.

**Релевантность по сегментам:**

| Сегмент | Бронирование | Обоснование |
|---------|-------------|-------------|
| Dining (ресторан, кафе) | Ключевая ценность | Столик, ожидание, обслуживание |
| Entertainment (бар, караоке, боулинг) | Частично (VIP-зоны, компании) | Вечерний досуг, предварительная запись |
| Quick (кофейня, пекарня, фаст-фуд) | Не нужно | Формат "взял и ушёл" |

### 6.2 Высокоуровневый flow

Пользователь нажимает «Забронировать» → выбирает дату, время, гостей → партнёр подтверждает/отклоняет → пользователь получает статус. Детальная спецификация — задача для отдельного Discovery Directive.

## 7. Масштабирование (видение, не текущий scope)

**Контент-направление.** Раздел «Новости» (один из главных навигационных баров): обзоры заведений, гастрономические маршруты, подраздел с акциями и новостями про них. Scope: Horizon 3-4, когда появится бюджет на репортёров.

**Расширение платных инструментов.** Push-уведомления "Избранным", расширенная аналитика (сравнение с конкурентами), участие в тематических подборках.

На вашей карточке никогда не будет рекламы конкурентов. А за доступную подписку — акции для привлечения клиентов и бронирование столиков прямо из приложения.»

## 9. Влияние на Horizon 2

### Приоритет 1 — Ядро ранжирования и claiming

- **Байесовский средний** (§2.5): заменить двухэшелонную сортировку на `weighted_rating = (v×R + 5×3.5)/(v+5)`, удалить `twoTierRating` и `RATING_SORT_MIN_REVIEWS`, применить ко всем query paths (§2.7)
- **Дефолтная сортировка** (§2.1): расстояние при геолокации, рейтинг без геолокации
- **Баг-фикс "тихий Минск"** (§2.8): `searchWithoutLocation()` + выбор города, убрать ложные расстояния
- **Отображение review_count** (§2.6): рядом с рейтингом на карточке (паттерн Booking.com) + pluralization
- Completeness score: логика расчёта → `base_score`, пересчёт при UPDATE
- Исправление ELSE 0 → ELSE 5 в price sort
- Обновление тайбрейкеров в `buildOrderByClause()` — с учётом двух режимов: с координатами и без (§2.3)
- Claiming: миграция (claimed_at, is_seed), admin-эндпоинты (claim + upgrade-to-partner)
- Партнёрский дашборд: completeness score (прогресс-бар)

## 10. Открытые вопросы

1. **Цена подписки** — конкретная стоимость с учётом белорусского рынка. Определяет количество тарифов (§4.4). Отдельная сессия.
2. **Правовые аспекты seed-данных** — использование реальных фото/описаний. Анализ практик рынка. Отдельная сессия.
3. **Бронирование** — детальная спецификация (UX, API, статусная модель). Отдельный Discovery Directive.
4. **Периоды подписки** — неделя и месяц, или только месяц?
5. **Claiming UX** — как партнёр узнаёт, что его заведение на платформе? Пока: invite-link от Координатора.
6. **Рефакторинг query paths** — единый query builder для всех режимов поиска. Не блокирует Horizon 2, но снижает риск рассинхронизации.
7. **Калибровка C (презумпция среднего)** — текущее значение 3.5 — консервативная константа. При росте базы и реальных отзывах пересмотреть: привязать к `AVG(average_rating)` или оставить фиксированным. Мониторить распределение рейтингов.
8. **Дизайн бейджа акции** — конкретный визуал для карточки и пина. Согласовать с дизайн-системой (theme.dart).
9. **UX выбора города** — дизайн экрана выбора города при отключённой геолокации. Сохранять выбор между сессиями или запрашивать каждый раз? Учесть сценарий: пользователь путешествует между городами.
